
## 一、产品概述

### 1.1 命名

**Sibylla**——寓意：一个拥有全局洞察力的 AI 系统，为团队提供判断与决策支持。

### 1.2 一句话定位

Sibylla 是一个以 AI 共享上下文为核心的团队知识协作平台——让团队中每一个人的 AI 都拥有整个团队的完整记忆。

### 1.3 核心差异

当前市场中，Notion 的 AI 只能看到当前页面，Obsidian 没有团队能力，Cursor 只为代码设计，飞书/Confluence 的 AI 无法深度理解项目全局。Sibylla 的 AI 能访问 workspace 内所有文档、所有历史、所有人的工作上下文，并基于此产出高质量结果——这是 **团队共享大脑** 的真正实现。

### 1.4 设计哲学

**文件即真相（File as Source of Truth）。** 所有内容以 Markdown 、 CSV 、JSON等文件形式存储在本地文件夹中。没有私有数据库格式，没有锁定。用户随时可以用任何文本编辑器打开自己的数据。

**AI 高度参与，人类最终决策。** AI 看到一切、分析一切、建议一切，但不自动执行任何不可逆操作。人类审批后才生效。

**Git 存在但不可见。** 底层用 Git 实现版本控制与多人同步，但用户界面中不出现任何 Git 术语。

**文件级协作。** 协作的最小单位是文件。不做实时协同编辑，不做块级锁定。这是对人类工作流和 AI 调用友好度的最大公约数。

**记忆即演化。** AI 不仅响应当前对话，更持续学习团队的工作模式、决策历史和知识积累。通过三层记忆系统（原始日志 → 精选记忆 → 归档），AI 能够理解项目演化轨迹，提供更有洞察力的建议。

### 1.5 目标用户

- 第一优先级：Crypto 远程办公团队（分布式、异步协作、对 Token 激励天然接受）。
- 第二优先级：AI 创业团队（对 AI 深度集成有强需求、早期采用者）。
- 第三优先级：自媒体团队（内容创作密集、多人协作、非技术用户为主）。
- 第四优先级：学生团体（价格敏感但传播力强、校园场景）。

Electron 桌面客户端 + 轻量云端服务。客户端承担编辑、本地存储、Git 操作、本地搜索。云端承担用户认证、Git 托管、通知推送、语义搜索索引、AI 模型网关、积分账本。

### 2.3 数据流

用户编辑文件 → 本地自动保存 → Git 抽象层自动 commit → 自动 push 到 Sibylla Git Host（或用户自带 GitHub）→ 云端触发语义索引更新 → 其他团队成员的客户端自动 pull → 本地文件更新。AI 对话时：用户提问 → 客户端组装上下文（Spec + 语义搜索结果 + 手动引用文件）→ 通过 AI Model Gateway 发送到模型 → 返回结果 → 如涉及文件修改则显示 diff 预览 → 用户确认后写入本地文件 → 触发同步流程。

## 四、核心模块与子功能清单

### 模块 1：文件系统与存储

这是 Sibylla 的地基层。所有数据以明文文件存储在本地文件夹中，保证透明性和可迁移性。

**1.1 Workspace 管理。** 创建 workspace（在本地创建文件夹并初始化标准结构）、打开已有 workspace、切换 workspace、workspace 设置面板（名称、描述、图标、默认 AI 模型、积分策略等）。

**1.2 文件树浏览器。** 树状目录展示（支持折叠/展开）、文件和文件夹的新建/重命名/移动/删除、拖拽排序与移动、文件类型图标区分（Markdown、CSV、图片、PDF 等）、文件状态标记（已同步/同步中/有冲突/待审核）、右键上下文菜单（常规文件操作 + "发送给 AI 分析"等快捷项）、文件夹级可见性标记（用于权限隔离）。

**1.3 文件导入与转换。** 拖拽导入文件夹或文件、Word (.docx) → Markdown 自动转换、Excel (.xlsx) → CSV 自动转换、PDF → Markdown 自动转换（OCR + AI 结构化）、Notion 导出包（Markdown + CSV）直接导入、Google Docs 导出包导入、保留原始文件副本在 `assets/` 中作为备份。

**1.4 附件管理。** 图片/PDF 等二进制文件存储在 `assets/` 目录、Markdown 中通过相对路径引用、粘贴图片自动保存到 `assets/` 并插入引用、附件去重（相同文件不重复存储）。

---

### 模块 2：WYSIWYG 编辑器

底层存储 Markdown，表层呈现富文本。用户不需要学习 Markdown 语法即可使用，而技术用户可以随时切换到源码模式。

**2.1 富文本编辑。** 标题（H1-H6）、粗体/斜体/下划线/删除线、有序列表/无序列表/任务列表（checkbox）、引用块、代码块（带语法高亮）、分割线、内联代码。

**2.2 高级内容块。** 表格（可视化编辑，底层存储 Markdown 表格或关联 CSV 文件）、数学公式（LaTeX 输入，即时渲染）、Mermaid 图表（流程图、时序图、甘特图）、嵌入图片（支持拖拽和粘贴）、文件内链接（`[[文件名]]` 语法，点击跳转）。

**2.3 编辑辅助。** 斜杠命令菜单（`/` 触发快捷插入）、工具栏（顶部常驻，常用格式化按钮）、快捷键体系（兼容主流编辑器习惯）、拖拽调整内容块顺序、字数统计与阅读时间估算。

**2.4 视图模式。** 编辑模式（默认，WYSIWYG）、阅读模式（纯渲染，不可编辑）、源码模式（直接编辑 Markdown）、大纲视图（按标题层级展示文档结构，点击跳转）。

**2.5 CSV 编辑器。** 表格视图（类 Excel 的网格编辑）、支持排序/筛选/搜索、单元格编辑、添加/删除行列、与 Markdown 文档中的表格互转。

---

### 模块 3：Git 抽象层

这是让非技术用户无痛使用版本控制的关键层。用户界面中不出现任何 Git 术语。

**3.1 自动保存与提交。** 文件修改后自动保存（防抖：停止输入 1 秒后触发）、保存时后台自动执行 `git add` + `git commit`、commit message 由 AI 自动生成（格式：`[成员名] 更新 文件名: 一句话变更摘要`）、批量修改合并为一次 commit（同一次操作涉及多个文件时）。

**3.2 自动同步。** 后台定时 push/pull（默认 30 秒间隔，可配置）、网络断开时本地正常工作，恢复后自动同步、同步状态在底栏实时显示："已同步 ✓" / "同步中 ↻" / "离线（本地已保存）" / "有冲突 ⚠"。

**3.3 冲突解决。** 冲突检测：pull 时发现同一文件被多人修改。冲突界面：左右对比视图（"你的版本" vs "对方的版本"），底部展示"AI 建议的合并版本"。操作选项：采用我的版本 / 采用对方版本 / 采用 AI 合并建议 / 手动编辑合并结果。解决后自动 commit 并 push。

**3.4 版本历史。** 文件级历史浏览：查看任意文件的所有历史版本列表（时间、修改者、AI 生成的变更摘要）。版本对比：选择任意两个版本进行 diff 对比。版本回滚：一键恢复到历史版本（创建新 commit，不破坏历史）。

**3.5 审核流程。** 管理员可以将特定文件夹标记为"需审核"。被标记文件夹中的编辑者修改会进入"待审核"队列（底层实现为 Git 分支）。管理员在"待审核"面板中看到变更摘要（AI 生成），可以批准、驳回、或附加评论。批准后自动合并到主分支。全程不出现 branch/merge/PR 术语，界面称之为"提交修改"和"审批修改"。

**3.6 Git 托管。** Sibylla 自托管 Git 服务（默认选项，面向非技术用户）：用户创建 workspace 时自动在云端创建对应仓库，无需任何配置。自带 GitHub/GitLab（高级选项，面向技术用户）：在 workspace 设置中填写远程仓库 URL 和凭证，支持 SSH key 和 Personal Access Token 认证。技术用户可以将仓库 clone 到本地用其他工具（如 VS Code）编辑代码，push 后 Sibylla 自动同步。

### 模块 4：AI 系统

Sibylla 的核心差异化能力。包含上下文引擎、对话界面、文档辅助、模式系统、模型网关五个子系统。

**4.1 上下文引擎（Context Engine）。** 这是"团队共享大脑"的技术核心。三层上下文自动组装：第一层"始终加载"——`CLAUDE.md`（项目宪法）+ 当前用户的 `personal/_spec.md` + 当前打开的文件内容。第二层"语义相关"——根据用户提问自动执行语义搜索，选取最相关的 5-10 个文件片段加入上下文。第三层"手动引用"——用户在对话中通过 `@文件名` 显式引用文件，被引用文件全文加入上下文。上下文来源标注：AI 回复时标注参考了哪些文件，用户可点击查看原文。上下文预算管理：根据所选模型的 token 限制，智能裁剪上下文（优先保留第一层和第三层，压缩第二层）。

**4.2 AI 主对话窗口。** 全屏对话界面，类似 ChatGPT/Claude 的体验。支持多轮对话、对话历史保存与搜索。支持 `@文件名` 引用 workspace 文件（自动补全）。支持 `@成员名` 引用特定成员的工作内容（加载该成员最近的文件变更）。文件修改能力：AI 建议修改文件时，显示 diff 预览（增删高亮），用户点击"应用"写入文件，或点击"编辑"修改后再应用。多文件操作：AI 可以一次性建议创建/修改多个文件，用户可以逐个审核或批量应用。对话导出：将对话内容导出为 Markdown 文件保存到 workspace。

**4.3 AI 文档侧边栏。** 在编辑器右侧呼出，面向当前正在编辑的文档。选中文字后激活快捷操作：改写（调整语气/风格）、扩写（补充细节）、缩写（精简表达）、翻译（支持多语种）、解释（对选中内容进行解读）、续写（从光标位置继续）。修改以 inline diff 形式嵌入文档，用户可接受或拒绝。全文操作：总结全文、检查语法和逻辑、生成大纲、按 Skill 规范检查格式。

**4.4 AI 模式系统。** 三种预设模式，本质上是叠加在 Skill 和 Spec 之上的 system prompt：

**Plan 模式**——定位为创意伙伴。行为：发散思维、提供多选项、方案对比、主动补盲点。典型场景：项目规划、功能设计、头脑风暴。输出倾向：结构化方案文档。

**Analyze 模式**——定位为分析师。行为：严谨、数据驱动、注重因果、标注不确定性。典型场景：竞品分析、数据解读、市场研究、可行性评估。输出倾向：分析报告。

**Review 模式**——定位为质量审查员。行为：批判性、逐项检查、指出问题与改进建议。典型场景：文档审查、方案评审、需求验收。输出倾向：检查清单和批注。

**自由模式（默认）**——不预设行为，AI 根据提问意图自动调整。用户也可在对话中随时切换模式。

**4.5 模型网关。** 用户在设置中配置多个模型的 API Key（Claude、GPT、DeepSeek、Gemini 等）。每种模式可绑定默认模型。对话中可随时切换模型。云端网关统一代理请求，记录 token 用量。支持 Sibylla 提供的共享 API（用户按量付费，无需自备 Key）和 用户自带 Key（BYOK）两种模式。

### 模块 5：Skill 系统

Skill 是 AI 的能力模块，本质是结构化的 prompt 模板，以 Markdown 文件形式存储。

**5.1 Skill 结构。** 每个 Skill 是一个 `.md` 文件，包含：Skill 名称与描述、适用场景说明、AI 行为指令（system prompt 部分）、输出格式模板（可选）、示例（可选）。

**5.2 Skill 作用域。** 团队共享 Skill：存储在 `skills/` 目录，所有成员可用。个人 Skill：存储在 `personal/[name]/_skills/`，仅个人可用。Skill 可以在 AI 对话中通过 `#skill名` 显式调用，也可以由 AI 根据任务自动匹配。

**5.3 预置 Skill 库。** 创建 workspace 时可选择初始 Skill 包：Crypto 项目技能包（tokenomics 设计、白皮书撰写、社区公告、链上数据分析等）、AI 创业技能包（PRD 撰写、技术方案设计、pitch deck 内容、用户研究等）、自媒体技能包（文案撰写、标题优化、SEO 分析、内容日历规划等）、通用技能包（会议纪要、周报、竞品分析、项目规划等）。

**5.4 Skill 管理。** 创建/编辑/删除 Skill（在编辑器中直接编辑 `.md` 文件）、Skill 索引（`skills/_index.md` 自动维护目录）、未来版本：Skill 社区共享市场。

---

### 模块 6：Spec 工作流

Spec 是 AI 的"行为准则"和"项目记忆"，定义了 AI 在本 workspace 中的行为规范和项目背景。

**6.1 Workspace 级 Spec。** `CLAUDE.md`——项目宪法，始终加载。内容包括：项目名称与简述、团队使命与价值观、核心业务逻辑概述、AI 行为通用准则（如"所有输出必须中英双语"）、重要决策历史摘要。`requirements.md`——当前需求文档。`design.md`——方案设计文档。`tasks.md`——任务清单与进度。这四个文件构成 workspace 的"项目大脑"。

**6.2 部门/组级 Spec。** 通过文件夹内的 `_spec.md` 实现。如 `docs/product/_spec.md` 定义产品组的工作规范。AI 在处理该文件夹内的文件时，自动加载对应的 `_spec.md`。不设独立的"部门"概念，直接用文件夹隔离，灵活度更高。

**6.3 个人级 Spec。** `personal/[name]/_spec.md`——个人工作偏好和上下文。内容包括：个人角色描述、负责的工作领域、个人偏好（如写作风格、语言习惯）、当前工作焦点。AI 在该成员的对话中自动加载。

**6.4 Spec 维护。** Spec 文件在编辑器中直接编辑（和普通文件一样）。AI 可以在对话中建议更新 Spec（如"这个决策应该记录到 CLAUDE.md 中"），用户确认后自动写入。`changelog.md` 由 AI 自动维护，记录 workspace 的重要变更事件。

---

### 模块 7：搜索系统

搜索是 AI 共享上下文能力的基础设施，直接决定 AI 能否"找到对的文件"。

**7.1 全文搜索（本地）。** 关键词搜索（支持模糊匹配）、文件名搜索、标签搜索（Markdown 中的 `#tag`）、搜索结果高亮与上下文预览、搜索速度要求毫秒级。

**7.2 语义搜索（云端）。** 文件每次 push 后，云端对变更文件进行 embedding。用户输入自然语言查询（如"上次讨论的会员体系方案"），返回语义最相关的文件和段落。搜索结果同时展示全文搜索和语义搜索的结果，按相关性混合排序。

**7.3 AI 对话中的自动搜索。** 当用户在 AI 对话中提到模糊引用（如"之前的竞品分析"），AI 自动调用搜索定位相关文件。AI 回复中标注引用来源，用户可以点击跳转。这是 Sibylla 与其他 AI 工具的关键体验差异——用户不需要手动找文件、手动粘贴内容。

**7.4 文件管理辅助。** 自动标签建议：AI 分析文件内容，建议添加标签。文件关联图谱：展示文件之间的引用关系（基于 `[[文件名]]` 链接和 AI 语义分析）。重复内容检测：提醒用户多个文件中存在重复或矛盾的内容。

---

### 模块 8：通知与信息流

异步协作的团队需要一个可靠的通知系统，确保重要信息不被遗漏。

**8.1 通知类型。** 文件变更通知：关注的文件被他人修改时通知（附带 AI 生成的变更摘要）。任务通知：任务被分配给你 / 你负责的任务状态变更。@提及通知：在文档或评论中被 @ 时通知。审核通知：你提交的修改被批准或驳回 / 有新的修改等待你审核。AI 洞察通知：AI 发现项目风险或需要你注意的事项。系统通知：workspace 设置变更、新成员加入等。

**8.2 通知中心。** 左栏入口，展示所有未读通知。按时间线排列，支持按类型筛选。点击通知直接跳转到相关文件或对话。支持标记已读、静音特定类型。

**8.3 推送渠道。** 客户端内通知（默认）。桌面系统通知（可配置）。后续版本：邮件、Telegram Bot、Discord Webhook 等外部推送。

---

### 模块 9：评论与讨论

文件级协作中，评论是最核心的交流方式。

**9.1 段落级评论。** 选中文档中的任意段落，点击评论按钮即可发起讨论。评论线程：每条评论可以回复，形成讨论线。评论中支持 @ 提及团队成员。评论中支持引用其他文件内容。评论数据存储为 Markdown 元数据（不污染原文内容），存储在 `.sibylla/comments/` 或与文件同名的 `.comments.json` 中。

**9.2 评论状态。** 开放（讨论进行中）、已解决（讨论结论达成，评论折叠但可展开查看）、AI 可以自动建议关闭已过时的评论。

**9.3 AI 参与讨论。** 用户可以在评论中 @AI 请求意见。AI 基于全局上下文给出回复，作为评论线程中的一条回复。典型场景："@Sibylla 这个方案和上次的竞品分析结论矛盾吗？"

---

### 模块 10：AI 项目管理

AI 深度参与项目管理，提供全局视角下的洞察和建议。人类做最终决策。

**10.1 任务管理。** `tasks.md` 为任务的 single source of truth，格式为结构化的 Markdown 任务列表。每条任务包含：标题、负责人、状态（待开始/进行中/已完成/已阻塞）、优先级、截止日期、关联文件。任务看板视图：将 `tasks.md` 渲染为可视化看板（按状态分列），支持拖拽变更状态。AI 辅助任务创建：在 AI 对话中讨论后，AI 建议将结论拆解为具体任务并写入 `tasks.md`。AI 状态自动追踪：AI 监测文件变更，自动建议更新任务状态（如检测到某 PRD 文件有实质更新，建议将对应任务标为"进行中"）。

**10.2 AI 日报/周报。** 每日/每周自动生成（可配置生成时间和频率）。个人报告（本人可见）：今日工作摘要（基于 commit 历史和文件变更）、任务进展、待办事项提醒。团队报告（管理员可见）：团队整体进度概览、各成员产出分布、本周关键变更汇总、风险预警（延期任务、长时间无进展的任务）、建议下周重点。报告以 Markdown 文件形式自动存储在 `personal/[name]/reports/` 或 `docs/reports/`。

**10.3 工作产出分析。** 分析维度（非"工时统计"而是"价值贡献"）：任务完成率与按时率、文档质量评分（是否符合 Skill 规范、完整性、清晰度）、协作响应速度（评论回复时效、审核处理时效）、对项目关键路径的贡献度、知识贡献度（创建了多少被他人引用的文档）。可见性规则：个人的完整分析仅本人和管理员可见。团队级汇总统计所有成员可见（匿名化或排行榜形式可配置）。分析结果用于积分计算。

**10.4 AI 决策建议。** 风险预警：AI 检测到项目风险时主动推送建议给管理员。资源建议：基于工作负载分析建议任务重新分配。决策辅助：当 workspace 中存在矛盾信息或未决策事项时，AI 整理所有相关信息并给出建议。所有建议以通知形式推送，管理员点击"采纳"或"忽略"。

---

### 模块 11：积分系统

MVP 阶段实现积分制，为后续 Token 化奠定基础。

**11.1 积分引擎。** 积分来源维度：任务完成（完成一项任务获得基础积分，按时完成有加成）、文档贡献（创建新文档、实质性编辑现有文档）、协作贡献（评论回复、审核处理、帮助他人解决问题）、质量加成（AI 评定文档质量高于基线时的额外积分）。权重可配置：管理员在 `tokenomics.md` 中定义各维度的权重。如自媒体团队可以将"文档贡献"权重调高，技术团队可以将"代码提交"权重调高。

**11.2 积分结算。** 结算周期可配置（每日/每周/每两周/每月）。AI 在结算时自动计算每个成员的积分，生成分配报告。管理员审核确认后正式记录。历史积分明细完全透明，所有成员可查看。

**11.3 经济模型预设。** 线性分配：按积分比例直接分配奖金/额度。二次方分配（Quadratic）：防止单人占比过大，鼓励广泛参与。固定池分配：每期固定总积分池，按比例分配。管理员可在预设基础上自定义参数。

**11.4 积分应用。** 绩效参考：积分历史作为绩效评估的客观数据输入。奖金分配系数：将奖金按积分比例分配。团队排行榜：可选开启，展示贡献排名。

**11.5 Token 化预留（P2 阶段）。** 数据结构设计时预留链上映射字段。积分账本采用不可篡改的 append-only 日志格式。后续可一键将积分映射为链上 Token。预留钱包地址绑定字段。

---

### 模块 12：权限与访问控制

**12.1 Workspace 角色。** Admin（管理员）：全部权限，管理成员、配置设置、审批修改、查看所有分析。Editor（编辑者）：创建/编辑文档、参与讨论、使用 AI。Viewer（查看者）：只读访问、可评论但不可编辑。

**12.2 文件夹级权限。** 管理员可以设置特定文件夹的可见性。例如 `docs/finance/` 仅对 Admin 和特定成员可见。权限配置存储在 `.sibylla/config.json` 中。AI 在为不同成员组装上下文时，自动排除无权限的文件。

**12.3 个人空间隔离。** `personal/[name]/` 目录默认仅本人和 Admin 可见。成员可以主动将个人空间中的文件"发布"到团队目录。

---

### 模块 13：MCP 外部集成

通过 MCP（Model Context Protocol）连接外部工具和数据源，扩展 AI 的能力边界。

**13.1 MCP 客户端。** Sibylla 内置 MCP 客户端，可连接外部 MCP Server。在 workspace 设置中配置 MCP Server 连接。AI 对话中可调用 MCP 工具。

**13.2 典型集成场景。** 连接 GitHub（读取 issue、PR 状态）、连接 Jira/Linear（同步任务）、连接 Slack/Discord（发送通知）、连接数据库（查询数据）、连接浏览器（网页搜索与内容抓取）。

**13.3 预置 MCP 配置模板。** 为目标用户群提供常用的 MCP 配置模板，降低配置门槛。

---

### 模块 14：迁移与导入

降低用户迁移成本，实现"倒入你的大脑"体验。

**14.1 一键导入。** 拖入文件夹即开始。支持混合格式批量导入。后台自动转换非 Markdown 格式。转换完成后 AI 自动建立索引。

**14.2 Aha Moment 引导。** 导入完成后引导用户打开 AI 对话。预设提问建议："我们项目目前的整体状况是什么？"AI 基于所有导入文件给出概览。这是用户第一次体验到"AI 真的读过我们所有东西"。

**14.3 渐进式迁移。** 不要求用户一步到位迁移所有内容。支持持续导入新文件。支持与原有工具并行使用一段时间。

---

### 模块 15：记忆系统

AI 的长期记忆能力，让 AI 从"每次对话从零开始"进化为"持续学习的团队伙伴"。通过三层分层存储实现知识的积累、提炼和检索。

**15.1 三层记忆架构。** 第一层"原始日志"——所有 AI 交互、文件操作、命令执行、决策讨论自动记录到 `.sibylla/memory/daily/YYYY-MM-DD.md`，append-only 模式不可篡改。第二层"精选记忆"——`MEMORY.md` 存储在 workspace 根目录，维持 8K-12K tokens，每次会话自动加载到 AI 上下文，由 LLM 智能提取高价值信息（用户偏好、技术决策、常见问题、项目约定）。第三层"归档层"——历史记忆按时间或主题归档到 `.sibylla/memory/archives/`，支持向量语义检索。个人空间内容不进入团队记忆。

**15.2 动态维护机制。** 心跳检查点：每 2 小时或每 50 条交互触发，LLM 分析最近日志并提取关键信息更新 MEMORY.md。预压缩冲洗：会话 token 达 75% 时自动将关键信息持久化，防止上下文丢失。决策日志：重要技术决策自动记录到 `.sibylla/memory/decisions/`，包含问题、方案对比、选择理由和实际结果。自愈优化：空闲时段自动检测重复内容、过时信息、孤立片段，维护记忆质量。

**15.3 检索与可视化。** 混合检索：结合向量语义检索（本地 embedding，all-MiniLM-L6-v2）和全文搜索（SQLite FTS5），使用 RRF 融合排序。查询 DSL：支持时间范围、标签、来源、操作者等多维过滤。记忆 Dashboard：右侧边栏展示 token 使用进度、质量指标（重复率/过时率/命中率）、最近活动。知识图谱：可视化展示主题、文件、决策、人员之间的关联关系。

**15.4 安全与隐私。** 个人空间隔离：搜索和检索自动排除其他成员的个人空间。敏感信息检测：日志写入前自动检测 API Key、密码等敏感信息并警告。记忆导出导入：支持团队间共享记忆，可选匿名化和脱敏。

## 六、开发路线图

### Phase 0：基础设施搭建

**目标：** 跑通最小技术栈。

**工作内容：** Electron 应用脚手架搭建（主进程 + 渲染进程 + IPC 通信框架）。本地文件系统读写封装（workspace 创建、文件 CRUD）。Git 集成基础（内嵌 isomorphic-git 或调用系统 git，实现 init/add/commit/push/pull）。云端服务基础框架搭建（用户认证 API、Git 仓库托管服务）。CI/CD 搭建（自动构建与发布 Electron 安装包）。

**里程碑：** 能够创建 workspace、编辑文件、自动 commit、push 到 Sibylla Git Host、另一台电脑 pull 下来看到变更。

---

### Phase 1：MVP 核心体验

**目标：** 实现核心体验闭环——"团队在 Sibylla 中协作编辑文档，AI 拥有全局上下文并产出高质量结果"。能够交付给 2-3 个内测团队使用。

**Sprint 1 —— 编辑器与文件系统**

- 文件树浏览器（目录展示、文件 CRUD、拖拽移动）。
- WYSIWYG Markdown 编辑器集成（基于 Tiptap 或 Milkdown 等开源编辑器）：基础富文本编辑（标题、列表、粗斜体、引用、代码块）、表格编辑、图片插入。
- 多 Tab 文件编辑。
- CSV 查看器（基础表格渲染）。
- 文件导入（Markdown 直接导入、Word/PDF 转换可用简单方案先行）。

**Sprint 2 —— Git 抽象层与同步**

- 自动保存 → 自动 commit（AI 生成 commit message 可先用简单模板，后续接 AI）。
- 自动 push/pull（定时同步）。
- 同步状态 UI（底栏状态指示）。
- 冲突检测与基础合并界面（三栏对比，先不接 AI 合并建议）。
- 版本历史浏览（文件级历史列表 + diff 查看）。
- Workspace 成员管理（邀请、角色设置）。
- Sibylla Git 托管服务上线（基础版，支持创建仓库、鉴权、push/pull）。
- 自带 GitHub 的连接配置。

**Sprint 3 —— AI 系统 MVP**

- AI 模型网关（云端代理，支持 Claude + GPT + Gemini + ...）。
- 上下文引擎 v1：始终加载层（`CLAUDE.md` + 当前文件）+ 手动 `@文件名` 引用。
- AI 主对话窗口：多轮对话、@ 引用文件、模式切换（Plan/Analyze/Review/Free）。
- AI 文件修改能力：diff 预览 + 用户确认写入。
- AI 文档侧边栏 v1：选中文字 → 改写/扩写/翻译。
- Spec 工作流：`CLAUDE.md` + `requirements.md` + `design.md` + `tasks.md` 自动识别并加载。
- Skill 系统 v1：`skills/` 目录识别、在 AI 对话中通过 `#` 调用 Skill、预置 Skill 包（1-2 套）。
- 全文搜索（本地关键词搜索）。
- 记忆系统基础架构：日志记录（append-only）、MEMORY.md 读写、文件锁机制、AI 交互自动记录。

**Phase 1 交付物：** 可安装的 Electron 桌面应用（Mac + Windows）。核心功能闭环：创建 workspace → 导入文件 → 编辑文档 → 自动同步 → AI 对话（带全局上下文）→ AI 修改文件。可交付内测。

### Phase 2：协作闭环

**目标：** 补齐团队协作所需的信息流通能力，让 Sibylla 从"能用"变成"好用"。

**Sprint 4 —— 语义搜索与上下文增强**

- 云端语义搜索服务上线（文件 embedding + 向量数据库）。

- AI 对话中的自动搜索（模糊引用自动定位文件）。
- 搜索 UI 优化（全文 + 语义混合结果、搜索结果预览）。
- 精选记忆提取：LLM 智能提取高价值信息到 MEMORY.md。
- 心跳检查点：每 2h 或 50 条交互触发记忆更新。
- 向量检索集成：记忆内容建立向量索引，支持语义检索。

**Sprint 5 —— 通知、评论、审核**

- 通知系统（文件变更、任务分配、@ 提及、审核）。

- 段落级评论系统（发起评论、回复线程、解决/重开）。

- 审核流程（文件夹标记"需审核"、待审核面板、批准/驳回）。
- Git 抽象层增强（AI 生成 commit message、AI 辅助冲突合并建议）。

**Sprint 6 —— 任务管理与日报**

- `tasks.md` 结构化解析与任务看板视图。
- 任务 CRUD（在看板中创建、拖拽状态变更，自动写回 `tasks.md`）。
