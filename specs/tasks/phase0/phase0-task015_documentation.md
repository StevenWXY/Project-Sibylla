## 任务描述

### 目标

汇编、整理并完善 Phase 0 阶段的基础技术体系文档。确保新加入项目的团队成员或 AI 代理，能凭借这些文档在 30 分钟内顺利搭建并跑通本地开发联调环境。沉淀项目当下的架构形态、模块划分、接口契约以及内部开发规范，为进入 Phase 1 提供坚实的指引支持。

### 背景

依据 `infrastructure-setup.md` 提出的“新开发者能在 30 分钟内完成环境搭建”的核心非功能需求，加之当前系统已包含了 Electron 前端、Node 后端、Gitea 服务等众多技术节点，通过进程间通信（IPC）与网络 API 进行组合协同。若缺乏严谨且明了的规范和使用指南，后续的协同开发将会面临严重的理解壁垒与错误重复发生。作为 Phase 0 的总结性工作，这份梳理工作至关重要。

### 范围

**包含：**
- 完善根目录的 `README.md` (包含最核心、极简的环境搭建引导路线)。
- 编写《开发者指南》(Development Guide, `docs/development/getting-started.md`)，详细说明各环境依赖、启动命令与数据库调试策略。
- 收集梳理并结构化当前系统已支持的 IPC 通道及通信格式字典。
- 收集梳理并结构化云端暴露的内部 RESTful API 及请求格式字典。
- 固化代码提交流转与测试规范（包含刚完成的 TASK014 中的 CI/CD 交互约定）。

**不包含：**
- 面向最终用户的《用户使用手册》（当前尚未有完整用户闭环功能，暂无必要）。
- 各个函数方法级别的内部设计细节（此部分通过源码的 JSDoc 或注释自我解释，不应过度写入文档）。

## 技术要求

### 格式与规范

- 文档采用 Markdown 格式组织，符合 `CLAUDE.md` 对文档的通用要求（如"注释语言使用英文，文档和 commit message 使用中文"）。
- 对于关键的复杂模块调用逻辑、系统时序或架构部署，需嵌入 Mermaid 图表予以直观表达，禁止使用过时格式或含糊不清的 ASCII 图。
- 保证跨文档引用的内部锚点和相对链接有效，确保良好的跳转阅读体验。

### 实现细节

#### 关键实现点

1. **环境快速搭建指南 (`README.md` 及相关引导)**
   - 清晰列出外部必须的前提依赖（如 Node.js 18+, Docker/Docker Compose, Git 等）及其配置建议。
   - 梳理出一套线性的命令链指令（例如从 `npm install` 到 `docker-compose up -d` 启动云环境，再到 `npm run migrate:up` 最后启动 Electron 的 `npm run dev`）。
   - 提供一个专用的“Troubleshooting（常见问题与排错）”小节，记录 Phase 0 开发中已知的常见环境阻碍点（如端口被占用、网络代理问题、Gitea 重置策略等）。

2. **IPC 与 API 字典维护 (`docs/development/api-ipc-reference.md`)**
   - **IPC**: 通过查阅 `sibylla-desktop/src/main/ipc/channels.ts` 及各 Handler，列出已支持的文件操作、Git 操作等命令，以表格化呈现（包括通道名、入参接口、返回类型或异常抛出说明）。
   - **API**: 梳理云端 `sibylla-cloud/src/routes/` 下的 RESTful API，输出标准的路由表清单，备注每个端点的鉴权要求和常见状态码。

3. **开发流转指南 (`docs/development/workflow.md`)**
   - 定义标准的 Git 分支开发模式（如功能开发从何拉取，如何提 PR）。
   - 说明在提交 PR 之前，开发者需在本地执行的验证操作（如本地 Lint 修复、通过哪些测试脚本），以呼应 CI/CD 设置。
   - 分享使用 `isomorphic-git`、Electron 跨进程开发等技术栈时的特殊约定与踩坑指南。
