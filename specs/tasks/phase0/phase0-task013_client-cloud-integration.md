## 任务描述

### 目标

验证 Phase 0 阶段所有客户端与云端基础设施的集成连通性，确保跑通“创建 workspace → 编辑文件 → 自动 commit → push 到云端 → pull 下来”的最小可运行链路，达到 Phase 0 核心里程碑。

### 背景

Phase 0 作为基础设施搭建阶段，已陆续完成 Electron 客户端基础（TASK001-003）、云端服务与 Git 托管（TASK004-007）、文件管理器与 Git 基础及同步机制（TASK008-012）。在此基础上，需要一个专门的集成测试环节来确保这些孤立的模块能够如设计般协同工作，暴露并修复跨模块协作问题。这直接呼应了项目"文件即真相"及自动同步原则的基础可行性验证。

### 范围

**包含：**
- 端到端（E2E）测试环境的搭建与配置
- 客户端与认证服务的集成联调测试（注册、登录、Token 刷新）
- 客户端与云端 Git 托管的集成联调测试（Clone、Push、Pull）
- 核心用户链路的自动化测试脚本编写（若可行）或标准的手工集成测试指南
- 测试文档与测试报告的生成规范

**不包含：**
- 单元测试（已在各自任务中完成）
- 性能压测（将由后续阶段负责）
- UI 自动化测试（本阶段侧重于核心数据同步及 API 链路验证，无需过度深入 UI 层面）

## 技术要求

### 技术栈

- **测试框架:** Vitest 或 Playwright（用于跨进程与网络交互的集成测试）
- **请求库:** axios 或系统原生 fetch（用于辅助查询云端状态）
- **环境配置:** dotenv（隔离测试环境配置）
- **测试环境:** 独立的 PostgreSQL 实例与 Gitea 实例（通过 `docker-compose.test.yml` 启动，隔离本地开发数据）

### 架构设计

集成测试环境需独立于开发与生产环境，避免污染真实数据。采用沙盒机制验证数据流向：
1. 启动测试专用的云端服务（包含 Auth, DB, Gitea）。
2. 在 Electron 主进程的测试桩（Test Harness）中模拟发起 IPC 调用。
3. 贯穿调用链路：IPC -> 主进程 FileManager/GitAbstraction -> Cloud API/Gitea。
4. 验证本地文件系统（`fs`）、本地 Git 状态与云端数据的最终一致性。

### 实现细节

#### 关键实现点

1. **测试环境脚手架**
   ```yaml
   # sibylla-cloud/docker-compose.test.yml
   # 仅暴露出测试专用端口，避免与开发环境（如 3000、5432）冲突
   ```
   - 编写启动和拆卸脚本（`setup/teardown`）：在测试套件执行前初始化数据库和 Gitea 测试配置，在测试结束后清理环境与临时目录。

2. **认证集成测试**
   - 模拟渲染进程发送 `ipc:auth:login` 类似请求（或直接调用核心服务）。
   - 验证主进程能正确通过 API 交互并获取、存储 JWT Token。
   - 验证在 Token 过期场景下，自动调用的 Refresh API 能无缝续期。

3. **核心数据同步链路测试**
   - **步骤 A**：模拟 Client 1 登录并创建 Workspace（验证云端与本地初始化的状态同步）。
   - **步骤 B**：模拟 Client 1 创建并编辑 Markdown 文件（验证自动保存、自动暂存与本地提交触发）。
   - **步骤 C**：触发 Client 1 的 Git Push 操作（验证与云端 Gitea 的权限校验及推送结果）。
   - **步骤 D**：模拟 Client 2 使用同一账号登录，Clone 该 Workspace（验证云端数据向异地拉取的操作）。
   - **步骤 E**：断言比对 Client 1 和 Client 2 本地文件系统的内容及 Git 历史记录的完全一致性。

## 验收标准

### 功能完整性

- [ ] 提供一键启动集成测试环境及运行用例的脚本（如 `npm run test:e2e`）。
- [ ] 认证链路测试覆盖率：成功登录、Token 存储、Token 刷新、未授权访问的 401 拦截。
- [ ] 同步链路测试覆盖率：Workspace 初始化同步、单文件变更推送、异地拉取更新全链路。
- [ ] 所有的集成测试在运行之间必须保持状态隔离，不因数据残留导致 Flaky Tests。

### 性能指标

- [ ] 整个集成测试套件完整执行时间应控制在 5 分钟以内。
- [ ] 测试环境数据库启动与结构初始化耗时 < 15 秒。

### 用户体验（针对开发者）

- [ ] 测试用例包含清晰的描述（`it('should ...')`）和直观的断言错误提示。
- [ ] 集成测试异常时输出的日志（Log）必须足够详尽，能够直接追踪请求生命周期。

### 代码质量

- [ ] 测试代码遵循 TypeScript 严格模式，无 `any` 滥用。
- [ ] 测试脚本符合项目的 ESLint 标准。
- [ ] 测试逻辑严格遵守 Arrange-Act-Assert（准备-执行-断言）模式。

## 测试标准

### 端到端联调验证
本任务自身旨在验证项目级别的联调，因此“跑通自身并生成 Green Build”即是最重要的测试标准。

**关键测试用例：**
1. 完整账号注册并自动登入。
2. 跨设备双边同步校验（并发编辑除外，本阶段不处理高级冲突解决）。
3. 无网络状态下的降级表现及网络恢复后的重试机制验证。
