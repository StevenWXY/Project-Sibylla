## 一、项目定义

Sibylla 是一个以 AI 共享上下文为核心的团队知识协作平台。
核心命题：让团队中每一个人的 AI 都拥有整个团队的完整记忆。
命名来源：Psycho-Pass 的 Sibyl System + 希腊神话女祭司 Sibyl。

## 二、设计哲学（所有技术决策的最终裁判）

1. **文件即真相**：所有用户内容必须以 Markdown/CSV 明文存储在本地文件夹中。禁止引入私有二进制格式或仅存于数据库中的用户数据。
2. **AI 建议，人类决策**：AI 可以看到一切、分析一切、建议一切，但不得自动执行任何不可逆操作。所有写入操作必须经用户明确确认。
3. **Git 不可见**：底层使用 Git 实现版本控制与同步，但用户界面中严禁出现 branch、merge、commit、pull、push 等术语。用"版本""同步""历史""审批"等自然语言替代。
4. **文件级协作**：协作的最小单位是文件。不实现实时协同编辑（CRDT/OT），不做块级锁定。
5. **记忆即演化**：AI 通过三层记忆架构（原始日志 → 精选记忆 → 归档）持续积累团队知识。日志不可篡改（append-only），精选记忆动态维护，归档支持语义检索。个人空间内容不进入团队记忆。

## 三、架构约束

- 客户端：Electron 桌面端为主力交付形态，后续补充轻量 Web 版（仅查看/评论/审批/AI 对话）。
- 云端服务：仅承担认证、Git 托管、通知、语义搜索、AI 网关、积分账本。不存储用户文档原文（原文仅存在于 Git 仓库和用户本地）。
- 本地优先：客户端在离线状态下必须可正常编辑和保存，联网后自动同步。
- Git 托管双模式：Sibylla 自托管（默认，面向非技术用户）+ 用户自带 GitHub（可选，面向技术用户）。
- 记忆系统：
  - 原始日志存储在 `.sibylla/memory/daily/YYYY-MM-DD.md`，采用 append-only 模式
  - 精选记忆存储在 workspace 根目录 `MEMORY.md`，维持 8-12K tokens，每次会话自动加载
  - 归档存储在 `.sibylla/memory/archives/`，支持向量检索
  - 心跳检查点：每 2 小时或 50 条交互触发记忆更新
  - 预压缩冲洗：会话 token 达到 75% 时自动持久化关键信息
  - 详细设计见 `specs/design/memory-system-design.md`

## 四、代码规范

### 通用
- 语言：TypeScript（严格模式，禁止 any）。
- 注释语言：代码注释使用英文，文档和 commit message 使用中文。
- 错误处理：所有异步操作必须有明确的错误处理，禁止静默吞掉异常。
- 日志：关键操作必须有结构化日志输出（who/what/when/result）。

### 客户端（Electron）
- 主进程与渲染进程严格隔离，通过 IPC 通信，渲染进程不得直接访问文件系统或 Git。
- 渲染进程使用 React + TailwindCSS。
- 状态管理方案由技术选型阶段确定后在此补充。
- 所有 AI 相关的 UI 交互必须有 loading 状态和错误兜底。

### 云端服务
- RESTful API 设计，路径格式：`/api/v1/{resource}`。
- 所有 API 必须有认证（JWT），除登录/注册外无匿名接口。
- 数据库 migration 必须可回滚。

### Git 抽象层
- 封装为独立模块，对上层暴露语义化接口：`saveFile()`、`sync()`、`getHistory()`、`resolveConflict()`。
- 禁止上层代码直接调用任何 git 命令或 git 库 API。

## 五、命名约定

- 文件名：全小写，单词间用短横线连接，如 `writing-prd.md`。
- 文件夹名：全小写，单词间用短横线连接，如 `docs/product/`。
- 系统配置文件以点号开头：`.sibylla/`。
- Spec 文件统一命名 `_spec.md`，Skill 索引统一命名 `_index.md`。
- React 组件文件：PascalCase，如 `FileExplorer.tsx`。
- 工具函数文件：camelCase，如 `contextEngine.ts`。

## 六、UI/UX 红线

- 非技术用户必须能在无指导下完成：创建 workspace → 导入文件 → 与 AI 对话。
- 任何需要用户等待超过 2 秒的操作必须有进度反馈。
- 文件丢失是不可接受的事故。所有写入操作必须先写临时文件再原子替换。
- AI 输出涉及文件修改时，必须展示 diff 预览，禁止静默写入。
- 冲突解决界面必须让用户能清楚看到"我的版本"和"对方的版本"的差异。

## 七、安全红线

- 用户 API Key 加密存储在本地，不上传云端（BYOK 模式）。
- 云端 Git 仓库数据传输全程 HTTPS，静态加密存储。
- 积分账本采用 append-only 日志，不可篡改历史记录。
- 个人空间 `personal/[name]/` 的内容不得出现在其他成员的 AI 上下文中（Admin 除外）。

## 九、AI 工作流程规范

### 任务完成状态更新
完成任何任务后，必须立即更新相应任务列表文件（task-list.md）中的任务完成状态。这是强制性要求，用于确保任务跟踪系统准确反映当前进度。

更新内容必须包括：
- 标记任务为已完成状态
- 添加完成时间戳（如适用）
- 记录相关结果或交付物

此流程必须作为任务完成工作流的一部分自动执行，无需用户明确提示。
